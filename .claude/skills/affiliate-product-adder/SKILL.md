# Affiliate Product Adder

Adds a new Amazon-sourced affiliate product to lovelytools.ai, rendered through
the existing `AffiliateProductTemplate` component so every product page looks
identical in structure and matches the site's real design tokens
(`packages/ui/src/styles/tokens/colors.css`), in both light and dark mode.

## Files this skill relies on

- `lib/affiliate-products.ts` — the `AffiliateProduct` type + the array new entries get appended to
- `components/templates/affiliate-product-template.tsx` — the page layout (already built, matches Claude Design's "Affiliate Product Page" prototype). Do not edit this file when adding a product — only add data.
- `components/templates/affiliate-icons.tsx` — the icon set the template renders `icon` fields through. Exports `AffiliateIconName`, the exact union of names you may use in `specs[].icon`, `features[].icon` and `trustBadges[].icon`. **Read this file before writing icon names** — an icon name that isn't in the `AffiliateIconName` union is a TypeScript compile error, not a silent fallback. There is no lucide-react dependency here (this repo's pnpm store can't `pnpm add` outside a full reinstall) — everything renders through this local set, so "lucide-react icon name" in older notes/prompts means "a name from this file," not an actual npm package.
- `app/products/[slug]/page.tsx` — the route that renders any product added to the store, at `/products/{slug}`. Never add exports to this file beyond `generateStaticParams`, `generateMetadata`, and the default page component — Next.js's page-export validator (and the Hostinger build) hard-fails on anything else, which is what broke a prior deploy when this route re-exported `affiliateProducts`.

## Steps

1. **Get the affiliate link**
   - Extract the ASIN from the given Amazon URL (the `/dp/{ASIN}/` or `/gp/product/{ASIN}/` segment).
   - If the URL has no visible ASIN (e.g. a `/hz/mobile/mission/` redirect link), open it in the browser tool and read the resolved URL's ASIN instead.
   - Note the Amazon domain (`amazon.co.uk`, `amazon.com`, etc.) from the URL.
   - Affiliate tag is `lovelytools-21`.

2. **Fetch the product data**
   Open the canonical product page (`https://www.{domain}/dp/{ASIN}`) with the browser tool. Dismiss the cookie-consent dialog first (click Decline/Accept) — it blocks reading the page otherwise. Extract:
   - Product title → split into `brand` (first word/brand name) and `name` (rest of the title)
   - Price (for reference only — the template always sends users to Amazon for current price, per the design; don't hardcode price on the page)
   - Rating and review count (from the stars/review-count line near the title) — this is what `score` and the trust-badge rating must be grounded in, not a guess
   - Main product image + up to 4 additional gallery images. Pull these with a JS snippet against `#landingImage`/`#imgTagWrapperId img` for the hero and `#altImages img` for thumbnails, then swap the `._AC_US100_` (or similar small-size) suffix in each thumbnail URL for `._AC_SX679_` to get a full-resolution version before downloading.
   - Bullet points / "About this item" → source material for `specs`, `features`, `pros`
   - Category breadcrumb on the Amazon page → informs `categoryLabel` (a reasonable guess is fine; the user will fix this once the real category exists). Set `categoryPath` to `/buyers-guide` — there is no real category route yet, and pointing it at a guessed path 404s.

3. **Download images locally**
   Save the product images under `public/products/{slug}/` (create the folder), numbered `1.jpg`, `2.jpg`, etc. — `1.jpg` is the hero. Reference them in the entry as `/products/{slug}/1.jpg` etc. Never hotlink Amazon's own image URLs — they can break or change. Use `curl -sL -A "Mozilla/5.0"` (no user agent gets blocked by Amazon's CDN).

4. **Write the copy**
   Fill in, in the user's/brand voice, matching the tone of the existing example entry (RYOBI Combi Drill — practical, specific, no hype-filler):
   - `tagline` — one punchy sentence
   - `description` — 1–2 sentences on the core problem/benefit
   - `specs` — 6–8 short label/value pairs from the listing, **each with an `icon` from `AffiliateIconName`** (e.g. `zap` for voltage/power, `gauge` for speed, `battery` for battery specs, `ruler` for dimensions)
   - `features` — 4–6 short feature callouts, **each with an `icon`** — these render as icon-in-circle badges, so pick something visually distinct per feature, don't reuse the same icon twice in one product if you can help it
   - `trustBadges` — 3–4 short stat callouts (e.g. "2 Batteries / Included, ready to go"), **each with an `icon`**
   - `pros` / `cons` — grounded in the actual listing and reviews, not invented
   - `bestFor` / `notIdealFor` — who this product suits
   - `score` — an honest 0–10 based on the listing's rating/review signal (do not inflate). The template converts this to a 5-star display (`Math.round(score / 2)`), so keep that conversion in mind — a 9.2 reads as 5/5 stars, a 7.0 reads as 3.5/4 stars.
   - `verdict` — 1–2 sentence summary
   - `faq` — 3–4 Q&As sourced from the listing's actual Q&A section or common bullet points. These render as a collapsed `<details>/<summary>` accordion automatically — no extra markup needed, just plain `{ q, a }` pairs.

5. **Generate the slug**
   Kebab-case of brand + core product name (e.g. `ryobi-combi-drill-starter-kit`). Check `allAffiliateProductSlugs()` in `lib/affiliate-products.ts` to avoid collisions.

6. **Append the entry**
   Add the new `AffiliateProduct` object to the `affiliateProducts` array in `lib/affiliate-products.ts`. Do not touch any other array entries. `images` is typed as a non-empty tuple (`[string, ...string[]]`) — always include at least one image or this is a type error.

7. **Verify**
   Run `pnpm dev` (or confirm the user already has it running) and check `/products/{slug}` renders correctly:
   - Both light and dark mode — the template pulls dark/light entirely from the site's existing theme toggle, no extra work needed.
   - Every `icon` field actually renders an SVG (open devtools, confirm no blank circles) — a typo'd icon name is a TypeScript error at build time, so this should be caught by `pnpm typecheck`/`pnpm build` before it ever reaches the browser, but double check visually anyway.
   - The FAQ accordion opens/closes on click.
   - It also appears on `/buyers-guide` (the listing page reads the same store automatically — nothing to wire up there).
   Then run `pnpm build` (or `npm run build` from repo root) once before handing back — this is a static-export site, and a page-export or type error here is exactly what breaks the live Hostinger deploy even when `next dev` looks fine locally.

8. **Report back**
   Tell the user the new page URL (`/products/{slug}`) and flag anything you had to guess (category label, score) so they can review it.

## What NOT to do

- Don't invent specs, pros/cons, or FAQ answers not grounded in the actual Amazon listing.
- Don't invent a `score` beyond what the listing's rating/reviews actually support.
- Don't modify `affiliate-product-template.tsx` or `affiliate-icons.tsx` per-product — all customization is data, not markup, so every product page stays visually consistent. If a concept genuinely needs a new icon, add it to the `AffiliateIconName` union and the `paths` map in `affiliate-icons.tsx` (matching its existing 24x24 stroke style), don't inline a one-off `<svg>` in the template.
- Don't use an icon name that isn't in `AffiliateIconName` — check the file first.
- Don't hardcode price — Amazon prices change; the template always routes to Amazon for current price.
- Don't skip the affiliate tag — every outbound link must go through `affiliateUrl()` in `lib/affiliate-products.ts`, never a raw Amazon URL.
- Don't add exports to `app/products/[slug]/page.tsx` beyond the standard Next.js page exports — see the file note above.
- Don't skip the `pnpm build` check in step 7 — `next dev` does not catch every error the production/Hostinger build does (page-export validation in particular is build-only).
